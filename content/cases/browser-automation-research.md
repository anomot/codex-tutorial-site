# 场景
- 需要用 Codex 配合浏览器操作完成一轮调研或产品验证。
- 目标是让调研过程可复用，而不只是人工随手看页面。

# 前置条件
- 已确定目标网站或目标页面。
- 已明确要回答的问题或验证点。
- 如涉及登录或表单，有合法访问权限。

# 背景
- 浏览器自动化调研不只是“打开网页看一眼”，而是围绕问题收集结构化证据。
- 对教程站来说，这类案例非常适合展示 Codex 如何把观察、记录、验证串起来。

# 输入材料
- 调研目标
- 目标 URL 或页面清单
- 需要验证的字段、流程或交互
- 记录模板

# 推荐 Prompt
```text
请基于浏览器操作能力，完成这次调研：
1. 先读取页面结构；
2. 执行指定交互；
3. 记录关键观察；
4. 保存截图或日志；
5. 总结发现与风险；
6. 给出下一步建议。
```

# 真实操作流程
1. 明确调研问题。
2. 打开目标页面并读取结构。
3. 执行关键交互或流程。
4. 记录页面行为、控制台信息、截图与结论。
5. 输出结构化调研结果。

# Codex 操作步骤
- 读取页面文本和结构。
- 执行最小交互。
- 收集证据。
- 总结结论。
- 给出后续动作建议。

# 人工检查点
- 观察是否遗漏关键交互。
- 检查截图和结论是否一致。
- 检查是否把临时现象误当规律。

# 核心观点
- 浏览器调研的重点不是自动点击，而是结构化记录。
- Codex 在这里的价值，是把调研流程标准化和可复用化。

# 可复用知识
- 问题 → 页面 → 交互 → 证据 → 结论，是固定闭环。
- 截图和日志比口头印象更可信。

# 最佳实践
- 调研前先写清楚问题。
- 优先记录证据，再写结论。
- 调研结果要能被二次复查。

# 常见坑
- 边看边想，最后没有结构化输出。
- 只记录结论，不保留截图和过程。
- 把单次现象当成长期规律。

# 测试与验证方法
- 检查输出是否覆盖调研目标。
- 检查是否保留截图、关键步骤和结论。
- 让另一个人根据结果复查一次。

# 输出结果
- 一份结构化浏览器调研报告。
- 一条可复用的浏览器调研 Prompt。
- 一套调研记录模板。

# 复盘
- 这类案例适合展示 Codex 在“研究型工作流”上的价值。
- 如果结论质量差，通常不是工具问题，而是调研目标定义不清。

# 可复用模板
```text
请先不要下结论，先把页面结构、关键交互、截图证据和观察记录列出来，最后再总结判断。
```
