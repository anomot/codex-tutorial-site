# 适用场景
- 你要把一个模糊需求整理成可执行任务。
- 你希望 Codex 先理解边界，再开始修改。
- 你需要最后能验收：改了什么、怎么验证、还有什么风险。

# 核心观点
- done-when 不是一句“功能完成”，而是一组可观察、可测试、可验收的条件。
- 验证写“怎么检查”，done-when 写“检查到什么结果才算完成”。
- 复杂任务要把背景、当前行为、期望行为、执行要求和输出格式写清楚。

# 可复制模板
```text
# Codex Task Prompt

## 角色

你是一个谨慎的资深工程师。请优先保证正确性、可维护性和最小必要改动。

## 目标

{{一句话说明任务目标}}

## 背景

{{为什么要做这个任务，当前遇到什么问题}}

## 上下文

相关文件或目录：

- {{file_or_dir_1}}
- {{file_or_dir_2}}
- {{file_or_dir_3}}

当前行为：

{{描述当前系统如何表现}}

期望行为：

{{描述修改后应该如何表现}}

## 约束

- 不修改 {{不可修改内容}}
- 不引入新的第三方依赖，除非先说明原因
- 不改变现有公开 API，除非任务明确要求
- 不做无关重构
- 保持最小可验证改动

## 执行要求

1. 先复述你理解的目标、上下文、约束和完成标准。
2. 给出简短修改计划。
3. 标出预计会修改的文件。
4. 开始修改。
5. 修改后运行验证。
6. 最后输出总结。

## 验证方式

请运行或说明以下验证：

- {{test_command}}
- {{build_command}}
- {{manual_review_steps}}

如果无法运行，请明确说明原因，并给出我可以手动执行的命令。

## 完成标准 done-when

任务只有在满足以下条件时才算完成：

- {{业务结果 1}}
- {{业务结果 2}}
- {{技术结果 1}}
- {{测试结果 1}}
- {{构建结果 1}}
- 没有无关文件变更
- 最终回复包含：
  - 修改文件列表
  - 关键变更摘要
  - 验证结果
  - 仍然存在的风险或注意事项

## 输出格式

请按以下格式输出：

1. 任务理解
2. 修改计划
3. 修改内容
4. 验证结果
5. 风险与建议
```

# 简单任务版本
```text
目标：
修复 {{问题描述}}。

上下文：
相关文件：
- {{file1}}
- {{file2}}

约束：
- 尽量最小改动。
- 不修改无关文件。
- 不改变现有 API。
- 不引入新的第三方依赖。

验证：
- 运行 {{test command}}。
- 运行 {{build command}}。
- 如无法运行，请说明原因并给出人工验证步骤。

完成标准 done-when：
- {{问题}} 不再出现。
- 相关测试通过。
- 构建通过。
- 没有引入新的 lint/type 错误。

执行方式：
请先复述你理解的目标、约束和验证方式，然后给出简短 plan，再开始修改。
```

# 复杂任务版本
```text
目标：
完成 {{功能/重构/架构调整}}，使系统支持 {{目标能力}}。

背景：
当前系统存在 {{问题/限制}}，希望通过这次修改达到 {{业务或技术收益}}。

上下文：
请重点查看：
- {{目录或文件 1}}
- {{目录或文件 2}}
- {{测试文件}}
- {{配置文件}}

当前行为：
{{描述现在代码如何工作}}

期望行为：
{{描述修改后应该如何工作}}

约束：
- 保持现有公开 API 不变，除非明确说明。
- 不修改数据库结构。
- 不引入新依赖。
- 不大规模重写无关模块。
- 优先最小可验证改动。

执行要求：
1. 先阅读相关文件。
2. 先输出你对任务的理解。
3. 给出修改计划。
4. 标注可能受影响的文件。
5. 修改代码。
6. 补充或更新测试。
7. 运行验证命令。
8. 最后总结修改内容、验证结果和风险点。

验证：
- 运行：{{test command}}
- 运行：{{build command}}
- 如有 UI 变化，请说明手动验证路径。

完成标准 done-when：
- {{核心功能 1}} 正常工作。
- {{核心功能 2}} 正常工作。
- 新增或更新测试覆盖主要逻辑。
- 所有相关测试通过。
- 构建无错误。
- 无无关文件变更。
- 最终回复包含修改摘要、测试结果和未解决风险。
```

# Bugfix 示例
```text
目标：
修复任务列表中点击“完成”按钮后，页面没有立即更新的问题。

背景：
当前用户点击完成按钮后，接口返回成功，但 UI 仍显示为未完成状态，必须刷新页面才会更新。

上下文：
相关文件：
- src/components/TodoList.tsx
- src/hooks/useTodos.ts
- src/api/todos.ts

约束：
- 不改变后端 API。
- 不修改 Todo 数据结构。
- 不引入新的状态管理库。
- 优先在现有 hook 内解决。
- 保持现有测试风格。

验证：
- 运行 npm test。
- 运行 npm run build。
- 手动验证：打开任务列表，点击一条未完成任务的“完成”按钮，确认 UI 立即变为完成状态。

完成标准 done-when：
- 点击完成按钮后，UI 立即更新为完成状态。
- 接口失败时不能错误地显示为完成。
- 已有任务列表功能不受影响。
- 新增或更新测试覆盖成功和失败两种情况。
- npm test 和 npm run build 均通过。
- 最终输出修改摘要、测试结果和潜在风险。
```

# Refactor 示例
```text
目标：
重构用户权限判断逻辑，将分散在组件中的权限判断统一迁移到 usePermission hook。

背景：
目前多个页面重复判断 user.role 和 feature flags，导致逻辑不一致，也不方便测试。

上下文：
重点查看：
- src/components/UserTable.tsx
- src/pages/AdminPage.tsx
- src/pages/SettingsPage.tsx
- src/hooks/usePermission.ts

约束：
- 不改变现有页面可见性规则。
- 不改变 user 对象结构。
- 不引入新依赖。
- 不大规模重写 UI。
- 每次修改保持可回滚。

执行要求：
1. 先找出重复权限判断逻辑。
2. 总结当前权限规则。
3. 给出重构计划。
4. 修改 usePermission。
5. 替换组件中的重复判断。
6. 补充测试。

验证：
- 运行 npm test。
- 运行 npm run build。
- 手动检查 AdminPage、SettingsPage、UserTable 的权限展示是否和之前一致。

完成标准 done-when：
- 重复权限判断逻辑统一进入 usePermission。
- 三个页面的权限行为和重构前一致。
- 新增测试覆盖 admin、普通用户、无权限用户三种情况。
- npm test 通过。
- npm run build 通过。
- 最终回复列出变更文件、重构前后差异、验证结果和风险点。
```

# 最佳实践
- 先写业务结果，再写技术结果，最后写测试和交付要求。
- 对 UI 任务写清楚目标视口、页面路径和人工检查方式。
- 对 bugfix 写清楚复现步骤和回归检查。
- 对 refactor 写清楚哪些行为必须保持不变。

# 常见坑
- 只写“功能完成”，没有可验收条件。
- 只写测试通过，没有业务行为标准。
- 把“运行测试”写成完成标准，而不是验证方式。
- 没有写不可修改范围，导致 Codex 顺手大改。
